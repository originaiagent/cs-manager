# L1（Lステップ併用開始）— トム向け手順書

作成: 2026-07-23 / 対象フェーズ: `docs/roadmap.md`「Lステップ脱却・LINE移行」L1（併用開始）
この1枚だけ読めば、トムがやる作業（申込み・確認・go-live判断）が分かるようにしてあります。エンジニア向けの技術設計は別ファイル（`docs/design/line-webhook-forward-receiver.md`）です。

---

## いま何をしようとしているか（30秒サマリ）

- 最終目標: Lステップを解約し、LINEの受信を cs-manager 直結にする。
- そのための最初の一歩が L1「併用開始」。
- LINE の「受信の入口（Webhook）」は1アカウントに1つしか設定できず、今はLステップが使用中。
- そこで**Lステップの「Webhook転送」オプション**（月5,500円）を使い、Lステップが受け取ったLINEの受信データを **cs-manager にもコピー転送**してもらう。
- これで「Lステップは今までどおり動く」まま、「cs-manager でも同じ受信を見られる」状態を作れる。返信は当面Lステップのまま（cs-managerは閲覧＋下書きだけ。二重返信を防ぐため）。

> ⚠️ この段階で一番の未確定ポイントは「Lステップが転送するとき、LINEの**署名（本物である証明）**をそのまま付けて送ってくれるか」です。付いていれば cs-manager は改修ゼロで受けられます（ケースA）。付いていない/形が変わる場合は、転送専用の合言葉（共有シークレット）で受ける改修が要ります（ケースB）。**どちらかは実際に1通受けてみないと分かりません。**だからこの手順書は「申込み → 1通受けて確かめる → 必要なら改修」の順番になっています。

## §A 申込みのときにトムが確認・記録すること（1枚）

Lステップの管理画面で「Webhook転送」オプションを申し込むとき、以下を**画面のスクショ or メモ**で残してください。これが cs-manager 側の受け口を「ケースA/ケースBどちらで作るか」を決める材料になります。

### A-0. オプションの基本情報（把握用 / 2026-07-23 管理画面で実確認済）
Lステップ管理画面「オプション一覧」に、Webhook転送は**2種類**あることを確認:
| オプション名 | 料金 | 現状 |
|---|---|---|
| **LINE Webhook転送　転送先数:1** | **月額5,500円（税込）** | 0/1（未利用） |
| LINE Webhook転送 ＋ API連携 | 月額11,000円（税込） | 0/1（未利用） |

- **まず狙うのは上の「転送先数:1」5,500円版**（これがcs-managerへ受信をコピー転送するのに必要な最小オプション）。
- **「＋ API連携」11,000円版**は、Lステップ側APIも叩ける上位版。**受信転送だけなら基本は5,500円版で足りる見込み**だが、後述A-2の「独自ヘッダー（合言葉）を付けられるか」がもし5,500円版で不可なら、この上位版が必要になる可能性がある。→ **申込む前に、まず5,500円版の「詳細」ボタンを開いて設定項目（特にヘッダー/認証欄）を確認**するのが安全（詳細を見るだけなら課金なし）。
- 「転送先1つ」でも問題なし。Lステップ自身は今までどおりLINEを処理し続け、そのコピーをcs-manager（＝この1つの転送先）へ送るだけ。承認まで約3営業日の想定。

### A-1. 転送先URL（cs-manager の受信口）に入れる値
- 転送先URL:
  ```
  https://cs-manager-chi.vercel.app/api/channels/line/inbound
  ```
  （↑ 本番URL。`APP_BASE_URL` と同じドメイン。変わっていたら担当クロードに一声）
- **URLに合言葉を付け足すのは非推奨**になりました（URLに秘密を入れるとアクセスログ等に残る恐れがあるため、設計レビューで不採用）。合言葉が要る場合は下記の「ヘッダー」で渡します。**URL欄はこの1本だけでOK。**

### A-2. 認証まわりの設定項目（★ここが一番の確認ポイント）
Lステップの転送設定画面（有料オプション申込後に見える）に、次があるかを確認してメモ・スクショ:
- [ ] **転送先URL** の入力欄（必ずあるはず）
- [ ] **カスタムヘッダー**（任意のヘッダー名＋値、または「Authorizationヘッダー」）を追加できる欄はあるか
      - **あれば** → cs-manager は「ヘッダー合言葉」方式で安全に受けられる（ケースB対応が一番きれい）。合言葉の値は担当クロードが用意。
      - **無ければ（URL欄しか無い）** → その旨を記録。公開情報では「URL欄のみ」しか確認できていないため、**この可否確認が最重要**。見当たらなければ**Lステップのサポートに「転送先へ独自ヘッダー／認証トークンを付けられるか」を問い合わせ**てほしい。
- [ ] **署名（x-line-signature）を転送に含める**旨の記載はあるか（ある/ない/記載なし をメモ）
- [ ] 転送のリトライ（失敗時の再送）・タイムアウトに関する記載はあるか

> 設定欄の有無を記録してくれれば、cs-manager側の受け口の作り方（ケースA=改修ゼロ / ケースB=ヘッダー合言葉）が確定します。**署名がそのまま届けばケースA（改修不要）**。まずは申込んで転送ON→テスト1通、が目的です。

### A-3. 申込み後、1通だけテスト受信を流す
- 申込み・転送ON後、トムのLINE公式アカウントに**自分でテストメッセージを1通送る**（または誰かに送ってもらう）。
- 担当クロードに「転送ONにした、テスト1通送った」と一声ください。**cs-manager 側に届いた生ヘッダ（署名の有無・本文の一致）を担当クロードが確認**して、ケースA（改修不要）かケースB（合言葉改修が要る）を確定します。
- ここまでで「LINEの受信が cs-manager に届く」ことの実証（L1のゴール）になります。

## §B L1完了までにトムがやる物理作業チェックリスト

順番に上から。★=トムにしかできない作業（課金・アカウント操作・go-live判断）。

- [ ] **★① Lステップ「Webhook転送」オプションを申し込む（月5,500円税込・課金発生）**
      - 全プランで利用可。申込み＝月額課金が始まる。ここが唯一のコスト。
- [ ] **★② 転送設定を入れる**（§A-1・A-2）。転送先URL＝上記の `/api/channels/line/inbound`。設定画面の項目を§A-2のチェックでメモ。
- [ ] **★③ 転送をONにする**（＝この時点からLステップが受信をcs-managerへコピー転送。Lステップ側の既存機能は止まらない）
- [ ] **★④ テストメッセージを1通送る**（§A-3）→ 担当クロードへ連絡
- [ ] ⑤（担当クロード作業）届いた署名の有無を確認しケースA/B確定 → 必要なら受け口を合言葉方式に改修（設計は codex 承認済みのものを使用）
- [ ] ⑥（担当クロード作業）cs-manager の LINE チャネルを受信ON（active化）＋ Core に必要な鍵を登録
- [ ] **★⑦ go-live 判断**: 「cs-manager でLINE受信を本番で受け始める」ことにトムがGOを出す（世に出すフラグON＝トム判断が必須）
- [ ] ⑧（担当クロード作業）本番でテスト受信が cs-manager の画面に出ることを一緒に確認（スモーク）
- [ ] **★⑨ L1完了の確認**: 「Lステップは今までどおり動いている」かつ「cs-manager でも同じ受信が見えている」の2点をトムが目視OK

### L1でやらないこと（二重返信を防ぐための線引き）
- cs-manager からのLINE**返信送信はしない**（この段階は閲覧＋下書きだけ）。返信は従来どおりLステップ。
- ↑ を守るため、L1中は **cs-manager 上でLINEの下書きを「承認」しない**でください（承認すると送信ジョブが拾って送ってしまう構造のため）。担当クロードが安全弁（送信オフ設定）を追加する案も検討中（roadmap の backlog）。

### 元に戻したくなったら
- Lステップの転送をOFFにすればいつでも即・元通り（Lステップ単独運用）。可逆なので気軽に試してOK。

## 困ったとき / 判断が要るとき
- 「オプションの画面にこの項目が見当たらない」→ その画面のスクショを担当クロードへ。受け口の作り方をこちらで合わせます。
- 課金・go-live 以外は基本こちら（担当クロード）で進めます。トムに戻すのは「①申込み（課金）」「⑦go-live判断」の2点が中心です。
