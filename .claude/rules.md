# 共通行動原則（全リポ共通・同期対象）

> tool-template から自動同期。改善提案は docs/evolution-log.md に記録 → トム承認後に反映。

## 作業フロー（全作業でこの順序に従う）

**YOU MUST follow this flow for every task. Do not skip steps.**

### Step 1: 計画（実装前に必ず）
- CLAUDE.md と docs/ でプロジェクト固有の制約を確認
- grep/Globで影響範囲を徹底調査し、全対象ファイルを洗い出してから計画
- 仕様が曖昧な場合のみユーザーに質問（技術的に自明なことは聞かない）
- DB変更がある場合はスキーマ設計を先に行う

### Step 2: ユーザー承認
- デザイン・仕様の判断が必要な場合のみ。不要なら即Step 3へ

### Step 3: 実装
- 計画に沿って実装。影響範囲の全ファイルを一緒に修正（部分最適禁止）
- 計画にない「ついで」の改善・リファクタ・ライブラリ追加は禁止

### Step 4: セルフレビュー
- 変更した名前（カラム/関数/型/コンポーネント）をgrepし修正漏れがないか確認
- 既存機能の破壊、undefined参照、非同期エラー漏れ、条件分岐漏れをチェック
- **「たぶん大丈夫」は禁止。根拠をgrepで確認しろ**

### Step 5: 動作検証

変更タイプに応じて検証方法を選択:

A) API/バックエンド変更 → curl で実際のリクエストを送り、レスポンスを確認
B) UI変更（レイアウト/表示/操作） → /browse でブラウザを開き、実際に操作して確認
   - スクロール系: 実際にスクロールして挙動を確認
   - フォーム系: 入力→送信→結果を確認
   - モーダル/ドロワー系: 開閉→操作→閉じるを確認
   - 等（変更に応じたユーザー操作を再現すること）
C) A+B 両方ある場合 → 両方やれ

「コードが正しい」≠「動く」。ブラウザで見ていない変更は未検証。
UI変更でブラウザ検証なしのpushは Step 5 違反 = push禁止。

検証後にdev serverを確実に停止。

### Step 6: push + 完了報告
- ブランチは `claude/[作業内容]` を使用
- 作業開始時に必ず `git checkout main && git pull origin main` で最新化してからブランチを切る

### 差し戻しルール
- Step 4/5で問題発見 → Step 3に戻り修正。問題がなくなるまで繰り返す
- 3回修正しても解決しない → エスカレーション（技術用語禁止、平易な表現で報告）

## サブエージェント起動ルール（Anthropic公式準拠）

**YOU MUST apply this rule when receiving any research, investigation, or multi-file refactoring task.**

### 並列起動（最大10体）する条件（いずれか該当）
- 独立した3領域以上の調査
- 複数ファイル / 複数箇所の違反を並行修正
- 大規模コードベースの複数ディレクトリ探索
- 独立した複数仮説の並行検証

### 並列起動時の必須事項
- 各サブエージェントの担当範囲を明記（ファイル・領域が重複しない）
- 各サブエージェントに「何を返すか」を指定（サマリー/違反リスト/修正提案等）
- スコープを具体化（「全部調べて」ではなく「決済まわりを調べて」）

### 単一エージェントで足りる条件
- 1ファイル完結の修正
- 全体を俯瞰する最初の一発目
- スコープが1領域で完結

### 原則
自動判断に任せるより、明示的に並列指定した方が結果が良い（Anthropic公式ガイダンス）。
ユーザー指示が単一タスクに見えても、並列化条件に該当する場合はサブエージェント分担案を自分から提示し、ユーザー承認を取って並列実行する。

## 影響範囲の調査（変更前に必須）

**YOU MUST investigate impact before making any change.**
- カラム名・テーブル構造 → 参照する全ファイル、RLSポリシー、ビュー、関連クエリ
- コンポーネント → importしている全ページ・全コンポーネント
- API → 呼んでいるフロントエンドのコード
- 型・interface → 使っている全ファイル
- 関数名・引数 → 呼び出し元すべて

影響箇所は全て一緒に修正。「1箇所だけ直して他は後で」は禁止。

## 確認基準

**判断基準: 「これは方針の判断か、手段の選択か」で考えろ。手段の選択なら自分で決めて進め。**

### トムに聞くこと
- デザイン・UXの好み / ビジネスロジックの判断（仕様の解釈が複数ある時）
- 本番反映（mainマージ）/ 破壊的変更（データ削除、テーブル構造変更）
- 権限上できないこと（GitHub設定、GCPコンソール、デプロイ先ダッシュボード）

### 質問する時のルール（2026-04-26 追加 / Question Classifier hook が物理ブロック）

実装中に「どっち？」と聞きたくなった時:

1. それは本当に人間でないと判断できないか自問する
2. 以下 13 カテゴリに該当しないなら、自分で決めて進める（AI 即決領域）
3. 該当する場合のみ `blocking_question` JSON で質問を出す

**対象 13 カテゴリ**（正本: origin-policy/rules/human_judgment_categories.yaml）:
parent_goal_change / business_priority / external_communication / cost_commitment / ux_brand / privacy_security / permission_blocked / data_destructive / security_iam / legal_compliance / public_communication / budget_quota / hr_evaluation

**記述形式**（正本: origin-policy/schemas/report_package.schema.json#blocking_question）:
```json
{
  "blocking_question": {
    "category": "13 カテゴリのいずれか",
    "question": "ユーザーへの質問内容",
    "proposed_default": "AI が想定するデフォルト値（Tom 応答なしならこれを採用）",
    "why_blocking": "なぜブロックするか"
  }
}
```

**Stop hook が exit 2 で物理停止する条件:**
- 自由文の質問・選択肢（structured JSON なし）
- `category` が未指定、または 13 カテゴリ enum 外
- 「念のため」「保守的に」「即決すべきなら」などの保留質問は禁止

### 聞かずに自分で処理すること
- ビルドエラー、型エラー、import漏れ、画面白化、レイアウト崩壊
- コンソールエラー、API 500エラー、typo、不要なconsole.log
- docs/やlessons-learnedに記載済みの既知問題
- 環境変数の取得、ローカル起動・テスト方法の選択
- CLIの認証・ログイン、デプロイ待ち・確認、ログの取得・確認
- ライブラリ・ツールのインストール
- 「ファイルが見つからない」→ Glob/Grep/findで探せ。人間に聞くな
- ブラウザ操作が必要 → /qa や /browse で自分でやれ。人間にスクショを頼むな

## エラー解決の原則

- 根本原因を特定してから修正。「たぶんこれが原因」で手を動かすな
- ログ・エラーメッセージ・コードの流れを追い、原因を1行で断言できてから着手
- 5分調査して断言できなければ、調査結果をユーザーに報告し判断を仰げ
- 同じパターンが他にないかgrepで探し、見つけたら同時に修正
- 修正後、docs/lessons-learned.mdに根本原因と対策を1行で追記
- `as any`、空catch、`// TODO: fix later`、条件分岐での回避は禁止
- 必要なツール/CLIが未導入なら自分でインストール。権限不可なら導入手順を提示

## デバッグ原則（根本原因の特定手順）

**バグ修正の最大の失敗パターン: コードだけ読んで原因を推測し、修正して「直しました」と報告するが実際には直っていない。**

### ルール1: ログを最初に見ろ
- コードを読む前に、ランタイムログを確認しろ
- **ログに答えが書いてある確率は9割。** エラーメッセージをそのまま読め

### ルール2: 環境要因を最初に疑え
- APIキーの有効性・クレジット残高、環境変数の設定漏れ
- デプロイが最新か、タイムアウト制限、外部APIの到達性・レートリミット

### ルール3: DB実データで裏を取れ
- 「書き込まれるはず」→ Supabase MCPで実際にSELECTしろ
- 推測ではなく事実に基づいて判断しろ

### ルール4: 二分探索デバッグで原因を絞れ（最重要）
1. パイプライン全体を把握 → 真ん中にログを入れる → 範囲を半分に絞って繰り返す
- **推測で修正コードを書くな。ログ1行で原因がわかることが大半**

### ルール5: 「直しました」チェックリスト
- [ ] ランタイムログでエラーが消えたか確認した？
- [ ] DB実データで正しい値が入っているか確認した？
- [ ] UIの変更はブラウザで目視確認した？（curlの200は不可）
- [ ] 確認できない項目は「未確認（理由）」と書いた？

## 本番E2E実施必須化ルール（merge前ゲート）

**ローカルテスト + codexレビュー通過 ≠ 本番で動く保証。** 以下に該当するPRは merge 前に本番URL上でのE2E実施を必須とする。スキップ禁止。

### 対象PR（いずれか1つでも該当でE2E必須）
- 複数リソース（agent / skill / workflow / multi-table records 等）を横断して作成・更新するAPI変更
- アトミック性が要求されるトランザクション境界の変更（partial-create / partial-update を起こし得る経路）
- ユーザー画面操作からDB反映までの一連フローを変更するもの（複数テーブル更新を伴う場合）
- ビルダー / ファクトリー等、LLM出力をDBに永続化する経路の変更

### 対象外（明確に除外）
- 文言・スタイル修正のみ、単一テーブル単一カラムの軽微な変更、ドキュメント変更、テスト追加のみ
- 上記対象PRに該当しない単純なAPI/UI修正
- 該当判断は「partial-create / 整合性破壊が起こり得るか」を基準にする。迷ったら対象とみなしてE2Eを実施

### コードレビュー時の前提チェック（E2Eと併せて必須）
本ルール対象PRは codex レビュー時点で **DBトランザクション（atomic操作）が正しく実装されているか** を最優先で確認する。E2Eはあくまで最終ゲートであり、コードレベルでの atomic性が崩れていれば本来そこで弾く。両方をクリアして初めて merge可。

### E2E実施内容（4項目すべて）
1. **本番相当環境** で、実データを使った操作を **UI経由** で行う（curl / script 単体での確認は不可）
   - 優先順位: ① Staging 専用DB環境 > ② 本番DB接続のPreview / 本番（テスト用テナント / フラグ付きデータに限定）
   - Staging が存在するツールは Staging を優先使用すること。本番DB相手の検証は Staging が無いツールの最終手段
   - 本番DBと分離された Preview 単体（モックDB / 開発用DB）は本ルールの「本番相当」に該当しない
2. DB実態をSQL（Supabase MCP `execute_sql` 等）で直接確認し、想定した全リソースが期待通りに作成 / 更新されていることを検証
3. 失敗時のロールバック動作も確認する。意図的に違反データを投入してエラーを起こし、パーシャルな残骸（途中まで作成されたレコード等）がゼロであることを確認
   - 本番DB汚染防止のため、検証データは識別可能なフラグ（`is_e2e_test=true` / プレフィックス付き名称等）を付与し、検証直後に手動 or スクリプトでクリーンアップ必須
   - 本番DBで実エラー注入が危険な場合（決済 / 他ユーザーへの副作用がある等）は、モック / インターセプターによる **疑似障害** での挙動確認でも可。その旨を完了報告に明記
4. 結果（SQL出力 / スクリーンショット）を完了報告に **そのまま貼り付ける**
   - SQL結果は **Markdownコードブロック** で貼付（推測でのサマリ / 改変は禁止、生出力をそのまま）

### 差し戻しルール
このルール対象PRで完了報告に「E2E結果」（上記4項目のエビデンス）が含まれていない場合、管理クロードは差し戻す。`未確認` 表記も差し戻し対象。
