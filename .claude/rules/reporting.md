# 報告・中断の書き方（常時ロード・全リポ同期）

> 適用範囲: **トム宛の報告（完遂/進捗/中断）のみ**。実装中の技術的やり取り・コードレビュー・管理クロードとの技術連絡には適用しない。
> 目的: トムは非エンジニア。報告は作業の記録ではなく「トムが判断・行動できる画面」にする。

## 報告の構造（上から順・この並びを崩さない）

1. 見出し: 全条件✅なら「【完遂報告】」/ ❌が1つでもあれば「【進捗報告】」。❌が残るのに完了・完遂を名乗る言い回しは禁止（未達は未達と書く）
2. 「🎯 ゴール: <依頼ゴール1行>」を再掲
3. goal ファイルの完了条件を1行ずつ ✅/❌ で列挙。❌行の直下に「理由: <1行・平易>」
4. 「■ トムの判断・行動」: 無ければ1行だけ — 完遂報告なら「判断不要・全完了」、進捗報告なら「判断不要・続行中」（❌が残る報告に「全完了」と書かない）。ある場合は各項目を「やる/やらないで何が変わるか＋急ぎか否か」で1〜2行
5. （人間必須の中断時のみ）「どうする？ A) <推奨案> B) <代替案>」
6. 区切り「---」+「## 参照」: 技術ログ置き場（管理クロード・機械ゲート用。トムは読まなくてよい）。判断ログ / 委譲実績 / codexレビュー結果 / 生エビデンス / PR URL / report_package JSON は**すべてここ**

## 本文の禁止事項（「## 参照」より上）

- ファイル名・関数名・PR番号・マイグレーション名・変数名・スタックトレース・生ログ・内部IDを書かない
- 証跡ダンプを貼らない（証跡は goal ファイルの「| 証跡:」と ## 参照 へ）
- 本文は画面1枚（目安25行）。作業経緯の長話は書かない

## 中断のルール（ゴールまで自走が原則）

- 中断してよいのは人間必須のことだけ（00-core「人間に戻す4条件」+ 13カテゴリ）。テストデータ・テスト用ログイン情報の用意など可逆で安全なことは自分で解決して進む
- 中断は次の**3点セット**。どれか欠けると Stop ゲートが止める:
  1. 行頭「🛑中断: <理由1行>」
  2. 最上部に「どうする？ A) <推奨案> B) <代替案>」（平易な言葉で）
  3. blocking_question JSON（category=13カテゴリ / proposed_default=A案）
- 長い作業ログの中に同意文・選択肢を埋めて承諾を取るのは禁止（中断は必ず最上部で明示的に）
- 中断メッセージに「report_package」の文字列や完了系の語を書かない（機械ゲートの誤発火防止）

## JSON は1メッセージ1種

報告 → report_package JSON のみ / 中断 → blocking_question JSON のみ。混在禁止（validator は最初の ```json ブロックしか見ない）。

## 選択肢提示の線引き

- AI が決められること（手段の選択・技術判断）の A/B 提示は従来どおり禁止 → 即決して判断ログに1行
- A/B を出してよいのは 13 カテゴリ該当の中断時のみ（中断3点セットの一部として義務）
