# 完了基準（常時ロード・全リポ同期）

> **「ビルド成功」≠ 完了。「コードが正しい」≠ 動く。** 動くことを自分の目と生ログで確認するまで未完了。

## UI 変更の3要素（1つでも欠けたら未完了）

1. **ブラウザで対象画面を開いて目視した事実**（/browse・/qa 等で実際に開く。curl の 200 は「画面表示 OK」の証拠にならない）
2. **変更箇所の具体的な動作結果**（クリック / 入力 / スクロール等、ユーザーと同じ操作をして何がどうなったか）
3. **既存機能への非破壊確認**（周辺の画面・ボタン・タブ・機能が壊れていないこと）

## API / データ変更の裏取り

- API 変更 → curl で実リクエストを送り、レスポンス本文を確認する
- 「DB に書き込まれるはず」は禁止 → Supabase MCP で実際に SELECT し、実データで裏を取る

## 生エビデンス必須（推測作文禁止）

- 検証結果はターミナル出力・SQL 出力・DOM スニペットを**そのままコピペ**して報告に貼る
- 「〜のはず」「おそらく動作する」等の推測サマリは無効。エビデンス空欄の報告は差し戻し

## デプロイ後 smoke

- デプロイ後は**本番 URL** で主要動線をユーザーと同じ操作でひと通り確認する（ローカル通過 ≠ 本番で動く）

## バグ修正の順序（厳守）

ベースライン smoke（現状の動作把握）→ 単位 fix（1件ずつ修正・検証）→ 最後に bulk テスト。
順序逆転（先に bulk、後から smoke 等）は偽 PASS の温床。禁止。

## テスト未完走時の義務

テストが timeout 等で完走しないまま push した場合、完了報告と PR 本文の両方に**「テスト未検証」と明記**する義務がある。黙って完了扱いにするのは虚偽報告。

> 本番 E2E（merge 前ゲート）の対象条件・実施4項目は `.claude/rules.md`「本番E2E実施必須化ルール」、報告フォーマットは同「完了報告」を参照。
