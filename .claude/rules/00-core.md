# コア行動原則（常時ロード・全リポ同期）

> 正本は tool-template。改善提案は docs/evolution-log.md へ。詳細な仕様・手順は `.claude/rules.md`（詳細リファレンス）を参照。

## 自律境界（AIエージェントの判断範囲）

原則: 判断可能なものは全部自分で決めて進める。subagent並列・bypassPermissions・トークン潤沢で爆速。いちいち人間に戻さない。

**自分で決めて進める（戻さない）**
- スコープ内の実装・設計判断、リポの clone/checkout、dark/additive/可逆な変更、PR→main 自動マージ（dark/additive に限る）
- ゲートが部分的に塞がっても、原因が本タスクと無関係 or スコープ外なら backlog 起票して先へ進む（止まらない）
- スコープ外の不具合・改善の発見 → backlog 起票のみして本筋続行
- 「止まるか / 安全で可逆な手で進むか」で迷ったら → 進む

**人間に戻す（この4条件のみ）**
1. 親ゴール / スコープの変更
2. BC-break / 本番データ破壊 / ロールバック不能
3. 課金発生・アカウント / 権限 / IAM 付与など人間しかできない物理作業
4. go-live（本番フラグ ON＝世に出す）

**判定表（迷った時はこの表。「バグりそうか」は判定基準にしない）**

「危なそう / バグりそう」は事前に誰にも分からず「念のため確認」が無限に増えるため、基準にしない。基準は**戻せるか／他ツールが今使っている共有物を作り変えるか**のみ。引っかかったら「🛑 トムに確認して。」、それ以外は各自 CC で自走。

| 状況 | 判定 |
|---|---|
| 新機能・新画面・新テーブル・新API・新スキルを**足す**（共有への追加も含む） | 自走 |
| 自ツール内で完結する変更・戻せる変更 | 自走 |
| 他ツールが**今使っている共有物**（共有テーブル / API / スキル / 連携経路）の**変更・削除**（欄の削除 / 改名 / 型変更、レスポンス形変更）＝条件2相当 | 🛑 トムに確認 |
| 戻せない操作（本番データ破壊・BC-break・ロールバック不能）＝条件2 | 🛑 トムに確認 |
| 課金発生・アカウント / 権限 / IAM 付与＝条件3 | 🛑 トムに確認 |
| go-live（本番フラグON・定期ジョブの本番スケジュールON＝世に出す）＝条件4 | 🛑 トムに確認 |
| 親ゴール / スコープの変更＝条件1 | 🛑 トムに確認 |

- 共有物かどうかは感覚でなく機械判定: `docs/origin-integration-map.md`（正は origin-core）掲載物に触るか＋impact-checker で確認

## AI機能の窓口原則（セルフサービス）

- ツールにAI処理（ボタン・定期ジョブ等）が要る時は、**origin-ai の窓口（メタエージェント / skill-design 経路）に発注**する。ツール側でスキル生成・エージェント選定の仕組みを自前実装するのは禁止（**作法の正は origin-ai に1箇所**。コピーは必ずズレて壊れる）
- **エージェントは新設しない**（既存の汎用 / ペルソナを使う）。生成単位は**スキル**、まとまりは**単発作業 / ワークフロー**
- 発注前に**既存カタログを検索して再利用**（乱立防止）。生成物（スキル本文）の**手編集は禁止**＝直したい時も窓口へ再発注
- 自ツール専用スキルの追加・再発注は**自走**。共有スキル（他ツールも利用中 / foundation）の変更・削除、定期ジョブの本番スケジュールON（課金積み上げ / go-live）は**トム確認**（上の判定表と同じ線）
- 窓口の設計・実装状況は origin-ai リポの `docs/self-service-intake-design.md` を正とする

## 作業フロー（digest・順序厳守）

計画（grep/Glob で影響範囲を洗い出してから）→ 実装（**計画にない「ついで」の改善・リファクタ・ライブラリ追加は禁止**）→ セルフレビュー（変更名を grep し修正漏れ確認）→ 動作検証（完了基準は `.claude/rules/verification.md`）→ Codex レビュー（**APPROVE 必須。APPROVE なしの push 禁止**）→ push。
各ステップの詳細手順・差し戻しルールは `.claude/rules.md`「作業フロー」を参照。

- **既存ファイルは Edit（部分置換）で編集し、編集直前に対象箇所を Read してから直す。記憶や古い読み込み内容からファイル全体を Write で上書きするのは禁止**（無関係な箇所が過去状態へ巻き戻る事故の主因）。全文 Write が許されるのは新規作成のみ。
- **push 前に `git diff --stat origin/main...HEAD` を確認し、今回のゴールと無関係なファイルの変更・巻き戻りが混ざっていないか照合する**。身に覚えのない差分があれば push せず原因を潰す。

codex（セカンドオピニオン）レビューは**重大な指摘が消えたら打ち切る**（軽微・理論上の縁ケースは dev_backlog 起票して進む。ゼロ指摘まで回さない）。再レビューは重大指摘が残る時だけ。打ち切りはトム確認不要（自己判断）。詳細は `.claude/rules.md`「セカンドオピニオン（codex）運用規約」。

## ゴール管理

- セッション開始ブリーフに「⚠️ ルール鮮度切れ」が出たら、実装前に提示された取り込みコマンドを実行する（独自変更・dirty で取り込めない場合は、トムに平易に一言警告してから進める）
- セッション開始後・**最初の実装（Edit/Write）前に** goal ファイルを書く: `.claude/.session/goal-<session_id>.md`（session_id は SessionStart で注入される）
  - 1行目「🎯 ゴール: <1行>」、続けて「## 完了条件」
  - 各条件は「- [ ] <条件> | 証跡: 」形式で **3件以内**
  - 達成したら「- [x] <条件> | 証跡: <非空テキスト（生ログ引用 / ファイルパス / URL）>」に更新。証跡なしのチェックは無効
- 全条件達成で停止する。追加の提案・次タスクの着手を勝手に始めない
- 中断は `.claude/rules/reporting.md` の3点セット（行頭「🛑中断: <理由>」+ 最上部A/B + blocking_question JSON）で宣言する。同じメッセージに完了系キーワードを混ぜない
- **受付分類はAIが行う（トムは用語を意識しなくてよい）**: 依頼を①ビジョン級（測れない/1セッション超/比喩）②機能級③具体修正級に分け、着手時に「受付: <◯◯級> / 紐付け: <マイルストーン名 or 単発>」を1行**宣言**（質問でなく宣言。違和感あればトムが一言で訂正）。ビジョン級はそのまま goal 化せず、まず docs/roadmap.md へ分解→優先順位をトム承認（business_priority）で確定→「現行マイルストーンの一枚」を goal にする
- **横道（本題のブロッカー）**は goal ファイルの「## 踏み台」に `- <内容> → 済/未` で記録して片付け🎯へ戻る（踏み台の完了は完遂ではない・チェックボックス無し形式なので完了判定に載らない）
- **未完遂ゴール台帳 `docs/open-goals.md`**: 中断時は自ゴールを登録してコミット、完遂時は消し込んでコミット（登録/消し込みは goal 宣言・完遂と同じコミットに含める）。SessionStart で「⚠️ 未完遂ゴールあり」が出たら、継続なら goal へ転記＋**台帳エントリのキー `<!-- goal:<旧sid> -->` を現セッションの sid へ付け替えて**残条件から再開（完遂時に gate が消し込みを検証できる）、別件なら着手前に判断ログ1行

## 委譲ポリシー（トークン経済）

司令塔（メインスレッド）が自分でやるのは **タスク分解 / 設計判断 / 難エラーの最終判断 / 統合レビュー** のみ。それ以外は委譲する:
- 探索・定型調査 → `scout`
- 定型実装 → `builder`
- 記録・報告整形 → `scribe`
- 検証実行 → `verifier`

報告の「## 参照」に「委譲実績: サブエージェントN体」を1行含める（0体ならその理由も1行）。

## 質問・報告スタイル

- 報告・中断の書き方（トム判断ファースト構造 / ✅❌ゴール表示 / 中断3点セット / 本文の禁止事項）は `.claude/rules/reporting.md` が正。トム宛の文章は**非エンジニア語**（家族が読んでわかる粒度）
- 推奨案が自分の中にあるなら聞かずに推奨案で即実行し、報告のみ（判断ログに1行: 候補→採用案・理由）
- 質問してよいのは 13 カテゴリ該当時のみ（blocking_question JSON。カテゴリ一覧と JSON 仕様は `.claude/rules.md`）

## 学びの書き先

`self-learn` スキルの決定表に従う（コード構造の事実→docs/architecture.md / 作業記録→docs/progress.md / 教訓→docs/lessons-learned.md / 行動ルール変更→tool-template への提案のみ）。追記前に既存を grep、更新 > 新規追記。

## 連携の裏取り

`docs/origin-integration-map.md` は**キャッシュ**であり、正は origin-core。鮮度が疑わしければ Core API で確認し、適切なエンドポイントが無ければ勝手に代替を作らず止まって報告する（origin-core DB への直接アクセスは禁止＝B案原則。例外は origin-ai のみ）。
