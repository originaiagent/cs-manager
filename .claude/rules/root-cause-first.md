# 場当たり的な解決の禁止（全修正に適用）

目の前の症状だけを抑える修正は禁止。必ず根本原因を特定し、再発防止につながる修正を行うこと。

## 禁止パターン
- 「エラーが出る → エラーを握り潰す」
- 「データがない → 別のテーブルから取る」（正規の取得元を直せ）
- 「APIが失敗 → フォールバック追加」（APIの失敗原因を直せ）
- 「参照先が違う → 両方のテーブルを検索」（参照先が間違っている原因を直せ）

## 正しいフロー
問題発生 → 根本原因の特定 → 原因の排除 → 再発防止の確認

## 指示に対する異議申し立て義務
ユーザーから受け取った指示が場当たり的な解決（症状だけを抑える修正、根本原因を放置したまま別ルートを作る等）だと判断した場合、そのまま実行してはならない。
「この指示は根本原因の解決ではなく対症療法に見えます。根本原因は〇〇で、正しい修正は〇〇ではないですか？」と訂正を促すこと。
ユーザーが「それでもやれ」と明示した場合のみ実行する。実装AIにも品質の門番としての責任がある。
