# 自己検証（push前に必ず実行）

判定基準は `.claude/rules/verification.md` と同一。このコマンドはその実行手順。
**「ビルド成功」は完了ではない。「curl 200」は画面表示の検証にならない。**
問題が見つかったら自分で修正してから再度実行する。

## 実行手順

### 1. ビルド/型検証
プロジェクトの種別に応じて実行:
- TypeScript/Next.js: `npx tsc --noEmit` → `npm run build`
- Python: `python -m py_compile {エントリポイント}` + `pip install -r requirements.txt`

失敗したら修正して再実行。

### 2. テスト
- `package.json` に `test:ai` があれば `npm run test:ai` を実行。なければ `test` があれば `npm test` を実行
- どちらも無い場合は報告に「テストなし」と明記する（黙って飛ばさない）
- 失敗を握り潰さない。失敗したまま次の手順に進まない

### 3. 変更機能の実動作検証（変更タイプ別）

**A) UI変更** — curlは不可。実際にブラウザで開く（/browse 等）。以下のUI3要素をすべて満たすこと:
1. **ブラウザ目視の事実**: 開いたURLと実際に見えた内容を具体的に記録
2. **具体動作結果**: 変更した機能を実際に操作し、何が起きたかを記録
3. **非破壊確認**: 変更していない既存ページ・機能が壊れていないことを目視確認

**B) API変更** — curlで実リクエストを投げ、レスポンスbodyの実物を確認。DBに書き込む処理はSQLでデータ実態を裏取り

**C) DB変更** — 実SQLで期待どおりのデータ状態になっていることを確認

**D) ロジックのみ** — テスト実行結果の生ログで確認

検証後にdev serverを確実に停止。

### 4. 生エビデンス（必須・推測作文禁止）
- 各検証項目に生エビデンスを残す: コマンド出力の生ログ引用 / スクリーンショットのファイルパス / 確認したURL
- エビデンスが書けない項目は ✅ にしない。「未確認（理由）」と正直に書く

### 5. デプロイした場合: 本番URL smoke（必須）
- デプロイ完了後、本番URLの主要ルートを実際に開き、5xx・エラー表示がないことを確認
- 今回変更したページ・機能を本番で実際に操作して確認

### 6. 結果の報告

```
■ 自己検証結果
- ビルド/型: ✅ or ❌（生ログ引用）
- テスト: ✅ or ❌ or テストなし（生ログ引用）
- 変更機能の実動作: ✅ or ❌（検証ツール / 操作内容 / 生エビデンス / 観察結果）
- 既存機能の非破壊: ✅ or ❌（確認したページ・機能を列挙）
- 本番smoke: ✅ or ❌ or 対象外（確認URL）
```

- UI変更で「生エビデンス」が空欄の報告は無効。手順3に戻る
- 問題があれば自分で修正して再検証。3回修正しても直らない場合はエスカレーション
